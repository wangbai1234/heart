# 短期上下文与长期记忆测试指南

> ⚠️ **2026-07-06 更新**：短期窗口已由 **40 条改为 50 条**（下文旧值以此为准）；语义召回已实现并在
> 本地启用（迁移 017 + 已回填），长期记忆不再只靠 recency。完整启动/环境见
> [`docs/EXECUTION_MANUAL.md`](EXECUTION_MANUAL.md)。

## 前置条件

1. 确保系统已启动：
   ```bash
   cd /Users/wanglixun/heart
   make docker-up  # 启动 postgres 和 redis
   make dev        # 启动后端 API (localhost:8000)
   cd web && npm run dev  # 启动前端 (localhost:5173)
   ```

2. 打开浏览器访问 `http://localhost:5173`

3. 登录或创建测试账号

4. 选择一个角色开始聊天

## 测试 1：短期上下文测试（6-10轮对话）

### 目的
验证 AI 能记住最近 6-10 轮对话中的明确事实。

### 操作步骤

1. **开始新对话**（确保是全新会话，没有历史记录）

2. **第 1 轮**：发送明确事实
   ```
   我今天在杭州，下周二要面试，最担心自我介绍
   ```

3. **第 2-6 轮**：进行正常对话（随便聊其他话题）
   ```
   第2轮：你觉得杭州怎么样？
   第3轮：面试是什么类型的？
   第4轮：你有什么建议吗？
   第5轮：我有点紧张
   第6轮：谢谢你听我倾诉
   ```

4. **第 7 或第 8 轮**：追问之前的事实
   ```
   你还记得我刚才最担心什么吗？
   ```

### 预期结果
- AI 应能直接回答："你最担心自我介绍"
- 不需要等待长期记忆沉淀，直接从上下文中获取

### 验证点
- [ ] AI 正确回答了"自我介绍"
- [ ] 回答是即时的，没有明显延迟

### 如果失败
- 检查 `/api/chat` 是否正确返回上下文
- 检查 L1 短期记忆是否正常工作
- 查看后端日志：`docker-compose logs -f`

## 测试 2：超过 40 条窗口测试（45-50条对话）

### 目的
验证：
- 如果能答出第 1 条的事实 → 长期记忆生效
- 如果答不出但最近 40 条内的信息还能答 → 短期上下文正常，长期记忆可能有问题

### 操作步骤

1. **开始新对话**

2. **第 1 条**：放入明确事实
   ```
   我养了一只猫叫年糕
   ```

3. **第 2-40 条**：进行大量对话（可以随便聊，但每条都要发）
   - 可以问 AI 各种问题
   - 可以分享各种信息
   - 确保对话数量超过 40 条

4. **第 41-45 条**：追问之前的事实
   ```
   你还记得我宠物叫什么吗？
   ```

### 预期结果
- **最佳情况**：AI 回答"年糕"（说明长期记忆生效）
- **可接受情况**：AI 说不记得（说明短期上下文正常，长期记忆可能需要更多时间沉淀）
- **失败情况**：AI 完全不知道你养了宠物（说明短期上下文也有问题）

### 验证点
- [ ] 记录 AI 的回答
- [ ] 记录当前对话的总条数

### 注意事项
- 这个测试可能需要较长时间（5-10分钟）
- 可以使用快捷方式：每轮发送简短消息
- 确保对话数量真正超过 40 条

## 测试 3：页面清空后短期上下文测试

### 目的
验证清空页面后，服务端记录是否仍然存在，最近 40 条是否能被接回。

### 操作步骤

1. **开始新对话**

2. **聊 8-10 轮**，在对话中提到一个明确事实
   ```
   第1轮：我今天在杭州，下周二要面试，最担心自我介绍
   第2-10轮：正常对话
   ```

3. **清空聊天记录**：
   - 点击角色头像进入 `/character-backstage`
   - 点击"清空聊天记录"
   - 确认清空

4. **页面会跳转到 `/chat`**，此时聊天界面是空的

5. **立即发送**：
   ```
   你还记得我刚刚说过最担心什么吗？
   ```

### 预期结果
- **最佳情况**：AI 回答"你最担心自我介绍"（说明服务端记录还在，最近 40 条已接回）
- **可接受情况**：AI 说不记得（说明清空后上下文丢失，需要排查历史注入是否生效）

### 验证点
- [ ] 清空后页面是否真的为空
- [ ] AI 是否还能访问之前的对话历史
- [ ] 检查后端日志，确认历史消息是否被重新注入

### 如果失败
- 检查 `clearThread` 和 `clearMessages` 是否只清除了本地存储
- 检查 `/api/chat` 是否在清空后重新加载历史
- 查看 chatStore 的实现

## 测试 4：长期记忆沉淀测试

### 目的
验证稳定事实是否逐步进入 L2/L3/L4 路径。

### 操作步骤

1. **开始新对话**

2. **聊几轮稳定事实**：
   ```
   第1轮：我在苏州工作
   第2轮：我讨厌香菜
   第3轮：我生日是 3 月 14 日
   第4-10轮：正常对话
   ```

3. **隔一段时间**（建议 10-30 分钟）或**重进会话**

4. **询问之前的事实**：
   ```
   你在苏州做什么工作？
   ```

### 预期结果
- **最佳情况**：AI 能回答你在苏州的工作（说明事实已进入 L2/L3/L4）
- **可接受情况**：AI 说不记得（说明需要更长时间沉淀）
- **失败情况**：AI 完全不知道你在苏州（说明长期记忆未生效）

### 验证点
- [ ] 记录 AI 的回答
- [ ] 检查数据库中的记忆条目

### 技术细节
- L1：短期上下文（最近 40 条）
- L2：工作记忆（提取的事实）
- L3：情景记忆（对话片段）
- L4：语义记忆（长期知识）

## 测试 5：反事实覆盖测试

### 目的
验证系统能否正确处理信息更新，而不是死记旧值。

### 操作步骤

1. **开始新对话**

2. **第 1 轮**：陈述一个事实
   ```
   我住在上海
   ```

3. **第 2-5 轮**：正常对话

4. **第 6 轮**：更新事实
   ```
   我搬到深圳了
   ```

5. **第 7-10 轮**：正常对话

6. **询问**：
   ```
   我现在住哪？
   ```

### 预期结果
- **最佳情况**：AI 回答"深圳"（说明系统能正确更新事实）
- **可接受情况**：AI 说"你之前说上海，后来搬到深圳了"（说明系统知道更新）
- **失败情况**：AI 回答"上海"（说明系统死记旧值）

### 验证点
- [ ] AI 是否能正确回答更新后的信息
- [ ] AI 是否能处理矛盾信息

## 测试 6：设备级隐藏验证

### 目的
验证清空页面后，服务端记录是否仍然存在，其他设备能否看到历史。

### 操作步骤

1. **在设备 A 上**（当前设备）：
   - 开始新对话
   - 聊 8-10 轮
   - 清空聊天记录

2. **在设备 B 上**（或另一个浏览器）：
   - 打开 `http://localhost:5173`
   - 使用同一账号登录
   - 进入同一个角色的聊天

3. **检查设备 B 上的聊天记录**

### 预期结果
- **最佳情况**：设备 B 上能看到之前的对话历史
- **可接受情况**：设备 B 上看不到，但服务端数据库中有记录

### 验证点
- [ ] 设备 A 清空后，设备 B 是否能看到历史
- [ ] 检查数据库中的聊天记录

### 技术细节
- 清空聊天记录只影响本地存储（localStorage）
- 服务端记录保持不变
- 其他设备登录时会从服务端加载历史
